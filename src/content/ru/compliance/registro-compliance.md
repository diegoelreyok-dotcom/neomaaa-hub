## 1. ОСНОВОПОЛАГАЮЩИЙ ПРИНЦИП: ЕСЛИ НЕ ЗАДОКУМЕНТИРОВАНО -- НЕ БЫЛО

В сфере compliance документация не является второстепенной административной задачей. Это доказательство того, что брокер выполняет свои обязательства. При аудите AOFA (или любого будущего регулятора) значение имеет не только то, что работа была выполнена правильно, но и то, что это можно доказать.

**Золотое правило:** Каждое решение по compliance должно иметь запись, включающую:
- Что было решено.
- Кто принял решение.
- Когда было принято решение.
- Почему было принято такое решение (обоснование).
- Какие доказательства были рассмотрены.

Если аудитор спрашивает "почему вы одобрили этого клиента?", а ответ -- "потому что он прошёл Sumsub", этого недостаточно. Правильный ответ: "Был одобрен [дата] [ответственным лицом], поскольку автоматическая верификация Sumsub подтвердила подлинность документа [тип], liveness check пройден успешно, скрининг санкций чист, и клиент не имеет индикаторов повышенного риска. Полная запись находится в [расположение]."

## 2. МАСТЕР-РЕЕСТР COMPLIANCE

### 2.1 Что это

Мастер-реестр Compliance -- это центральный файл, консолидирующий всю деятельность compliance NEOMAAA. Он состоит из нескольких листов/вкладок в одном файле (Google Sheets или Excel), каждый из которых посвящён определённому типу записей.

### 2.2 Структура файла

**Имя файла:** `NEOMAAA_Compliance_Register_2026.xlsx` (или эквивалент в Google Sheets)

**Листы/вкладки файла:**

| Вкладка | Содержание | Ответственный за ведение |
|---|---|---|
| Dashboard | Сводка метрик и счётчиков | Автоматически (формулы) |
| KYC_Decisions | Все решения по KYC | Susana |
| Sanctions_Screening | Все результаты скрининга с совпадениями | Susana |
| Suspicious_Activity | Обнаруженная подозрительная активность и SAR | Susana |
| Training_Log | Проведённые обучения | Susana |
| Complaints | Жалобы, связанные с compliance | Susana |
| Audit_Trail | Действия и значимые изменения | Susana |
| Config | Справочные списки и валидации | Susana (первоначальная настройка) |

### 2.3 Вкладка: Dashboard

Dashboard -- это сводный лист, использующий формулы для извлечения данных из остальных листов. Должен отображать:

| Метрика | Базовая формула | Периодичность |
|---|---|---|
| Всего одобренных клиентов (текущий месяц) | COUNTIFS в KYC_Decisions | Автоматически |
| Всего отклонённых клиентов (текущий месяц) | COUNTIFS в KYC_Decisions | Автоматически |
| Всего на проверке (ожидающие) | COUNTIFS в KYC_Decisions | Автоматически |
| Sanctions hits за месяц | COUNTIFS в Sanctions_Screening | Автоматически |
| False positives за месяц | COUNTIFS в Sanctions_Screening | Автоматически |
| True matches за месяц | COUNTIFS в Sanctions_Screening | Автоматически |
| Поданные SAR (квартал) | COUNTIFS в Suspicious_Activity | Автоматически |
| Случаи, ожидающие >24 ч (ТРЕВОГА) | COUNTIFS с условием по дате | Автоматически |
| Среднее время проверки | AVERAGE с фильтром | Автоматически |

### 3.2 Цветовая кодировка

| Цвет строки | Значение |
|---|---|
| Светло-зелёный (#C6EFCE) | Одобрен -- без инцидентов |
| Жёлтый (#FFEB9C) | На проверке / Retry / Ожидание |
| Оранжевый (#FCD5B4) | На EDD / Эскалирован на Principals |
| Светло-красный (#FFC7CE) | Отклонён |
| Тёмно-красный (#C0504D, белый текст) | Подтверждённое санкционное совпадение |

### 3.3 Правила использования

1. **Каждое решение по KYC порождает строку.** Независимо от того, автоматическое оно или ручное.
2. **Автоматические одобрения регистрируются пакетно.** В конце дня Susana экспортирует автоматические одобрения Sumsub и регистрирует их с Revisor = "Automatico".
3. **Ручные проверки регистрируются индивидуально** в момент принятия решения.
4. **Столбец "Razon_Decision" является обязательным.** Даже для одобрений: "Автоматическая верификация успешна. ID [тип], liveness OK, скрининг чист."
5. **Не удалять строки.** При наличии ошибки добавляется корректирующая запись, а не удаляется исходная.

### 4.2 Цветовая кодировка

| Цвет строки | Значение |
|---|---|
| Светло-серый (#D9D9D9) | Разрешённое ложное срабатывание |
| Оранжевый (#FCD5B4) | На анализе / ожидает разрешения |
| Тёмно-красный (#C0504D, белый текст) | Подтверждённое истинное совпадение |
| Светло-голубой (#BDD7EE) | PEP, одобренный с EDD |
| Жёлтый (#FFEB9C) | Негативные СМИ на оценке |

### 4.3 Правила использования

1. **Каждое совпадение порождает строку.** Если клиент генерирует совпадения по нескольким спискам, каждое совпадение -- отдельная строка.
2. **Поле "Analisis" должно быть подробным.** Недостаточно написать "false positive". Необходимо указать: "Ложное срабатывание. Имя частично совпадает, но дата рождения различается (клиент: 1985-03-15, список: 1962-08-22). Страна рождения различается (клиент: Колумбия, список: Ливия). Вывод: однофамилец, это не одно и то же лицо."
3. **Снимки экрана сохраняются** в папке, организованной по дате и ID клиента.
4. **Строки с true matches выделены красным и никогда не удаляются.**

### 5.2 Типы подозрительной активности (список валидации)

| Тип | Описание | Типичный уровень риска |

| Отчёт команды | Сотрудник сообщает о подозрительном поведении клиента | Варьируется |

### 5.3 Когда подавать SAR (Suspicious Activity Report)

SAR должен быть подан в AOFA, когда: [VERIFICAR CON ABOGADO]

1. Имеется обоснованное подозрение, что средства получены от преступной деятельности.
2. Имеется подозрение, что клиент использует счёт для отмывания денег.
3. Имеется подозрение в финансировании терроризма.
4. Клиент пытается обойти контроли compliance (structuring).
5. Обнаружено, что клиент предоставил ложную информацию в ходе KYC.

**Важно:** Решение о подаче SAR принимает Susana при консультации с Principals. SAR подаётся регулятору, клиент НЕ уведомляется. Уведомление клиента о существовании SAR является уголовным преступлением в большинстве юрисдикций (tipping off). [VERIFICAR CON ABOGADO]

| "Не могу вывести свои деньги" | "Мы рассматриваем вашу заявку. Вывод средств в определённых случаях требует дополнительной верификации." | "Ваши средства заморожены из-за санкций." |

**Абсолютное правило:** НИКОГДА не раскрывать клиенту внутреннюю информацию compliance. Ни отдел продаж, ни поддержка, ни кто-либо ещё, кроме Susana и Principals, не может предоставлять детали решений compliance.

## 8. AUDIT TRAIL

### 8.1 Структура листа "Audit_Trail"

| Столбец | Название | Тип данных | Описание |
|---|---|---|---|
| A | ID_Registro | Текст | Автоинкремент (AUD-2026-0001) |
| B | Fecha | Дата | YYYY-MM-DD |
| C | Hora | Время | HH:MM |
| D | Usuario | Список | Susana / Principals / Sistema | Кто выполнил действие |
| E | Accion | Список | См. список ниже | Тип действия |
| F | Detalle | Текст | Описание действия |
| G | Registro_Afectado | Текст | ID изменённой записи (KYC-xxx, SCR-xxx и т.д.) |
| H | Valor_Anterior | Текст | Значение до изменения (если применимо) |
| I | Valor_Nuevo | Текст | Значение после изменения (если применимо) |
| J | Razon_Cambio | Текст | Почему внесено изменение |

### 8.2 Действия, регистрируемые в Audit Trail

| Действие | Когда регистрируется |
|---|---|
| Изменение записи | Любое изменение существующей строки в любом листе |
| Изменение решения | Решение по KYC отменяется (например, с "отклонён" на "одобрен") |
| Доступ к конфиденциальным данным | Когда кто-либо обращается к документам конкретного клиента по нерутинной причине |
| Экспорт данных | Когда данные экспортируются из реестра |
| Изменение конфигурации | Модификация списков валидации, стран, правил |
| Эскалация | Каждый случай эскалации |
| Коммуникация с регулятором | Каждая коммуникация, отправленная в AOFA или полученная от AOFA |
| Изменение политики | Любое изменение в политиках или процедурах compliance |

### 8.3 Почему Audit Trail критически важен

При аудите регулятор хочет увидеть две вещи:
1. Что решения были правильными.
2. Что никто не манипулировал записями постфактум.

Audit Trail доказывает второе. Если запись была изменена, должна остаться запись о том, кто её изменил, когда и почему. Запись без audit trail утрачивает достоверность перед аудитором.
