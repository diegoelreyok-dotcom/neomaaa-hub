## 2. Процесс пополнения — пошагово

### 2.1 Общий поток

```
Клиент инициирует депозит в Личном кабинете
        |
        v
Выбирает метод оплаты
        |
        v
PSP (Payment Service Provider) обрабатывает транзакцию
        |
        v
PSP отправляет callback/webhook в Skale CRM
        |
        v
Skale регистрирует депозит и отмечает как "Pending" или "Completed"
        |
        v
[Если завершён] Автоматическое зачисление на счёт MT5 клиента
        |
        v
Клиент получает email-подтверждение
        |
        v
[Если > $5,000] Автоматическое уведомление Finance + Dealing (Pepe)
```

### 2.2 По типу метода

#### Карты (Visa/Mastercard)
1. Клиент вводит данные карты на портале (токенизированы PSP, мы не касаемся данных карты)
2. PSP выполняет 3D Secure если применимо
3. Транзакция одобрена или отклонена за секунды
4. Если одобрена: webhook в Skale → автоматическое зачисление на MT5
5. Если отклонена: клиент видит сообщение об ошибке с кодом

**Типичные проблемы с картами:**
- **"Транзакция отклонена"** → Банк клиента заблокировал транзакцию. Попросить клиента позвонить в банк и авторизовать международные покупки
- **"3D Secure не пройден"** → Клиент не завершил верификацию. Повторить попытку или использовать другой метод
- **Депозит одобрен, но не отображается в MT5** → Проверить в Skale, пришёл ли webhook. Если нет, эскалировать в Finance для ручного зачисления
- **Сумма отличается от запрошенной** → Проверить конвертацию валюты. PSP мог списать в локальной валюте

#### Банковские переводы (Wire/SEPA)
1. Клиент выбирает wire transfer на портале
2. Портал показывает банковские реквизиты NEOMAAA (банк, SWIFT, IBAN, ссылка)
3. Клиент должен указать **номер счёта MT5 как ссылку** — КРИТИЧНО
4. Клиент делает перевод из своего банка
5. Средства поступают на банковский счёт NEOMAAA (1-3 дня)
6. Finance верифицирует депозит по банковской выписке
7. Finance сопоставляет ссылку с клиентом в Skale
8. Finance зачисляет вручную в Skale → зачисление на MT5

**Типичные проблемы с wire:**
- **"Не указал ссылку"** → Попросить у клиента банковское подтверждение. Finance сопоставляет по имени + сумме
- **"Прошло уже 3 дня и не отображается"** → Попросить у клиента подтверждение SWIFT. Проверить, есть ли средства в транзите на банковском счёте
- **"Сумма не совпадает"** → Банки-посредники взимают комиссии. Зачислить нетто-сумму, информировать клиента

#### Крипто (USDT, BTC)
1. Клиент выбирает крипто на портале
2. Портал генерирует уникальный адрес кошелька (или показывает фиксированный адрес с memo/tag)
3. Клиент отправляет крипто со своего кошелька
4. Система мониторит подтверждения в блокчейне
5. После подтверждения (TRC-20: ~20 подтверждений, ERC-20: ~12, BTC: ~3): автоматическое зачисление

**Типичные проблемы с крипто:**
- **"Отправил в неправильную сеть"** → Если отправил ERC-20 на адрес TRC-20 (или наоборот), средства потеряны. Решения нет. Информировать клиента с эмпатией
- **"Отправил сумму, но не отображается"** → Попросить TX hash. Проверить в блокчейн-эксплорере (tronscan.org, etherscan.io). Если подтверждено, но не зачислено, эскалировать в Finance
- **"Отправил с биржи, которая не показывает мой кошелёк"** → Попросить у клиента скриншот транзакции с биржи + TX hash
- **Конвертация крипто в USD** → Зафиксировать обменный курс на момент зачисления

#### Локальные методы LATAM (PIX, PSE, SPEI, OXXO, Nequi, Yape, Mercado Pago)
1. Клиент выбирает локальный метод на портале
2. Локальный PSP генерирует QR-код (PIX, Yape, Nequi) или перенаправляет в банк (PSE) или показывает код оплаты (OXXO)
3. Клиент оплачивает в банковском приложении или в точке продаж (OXXO)
4. Локальный PSP подтверждает оплату → callback в Skale → зачисление на MT5

**Типичные проблемы с локальными методами:**
- **OXXO: "Оплатил, но не отображается"** → OXXO может занимать до 24 часов. Попросить чек оплаты, проверить с PSP
- **PIX: "QR истёк"** → Создать новый депозит. QR-коды PIX истекают через 30 минут
- **PSE: "Мой банк не отображается"** → Проверить с PSP, подключён ли банк. Предложить альтернативный метод
- **Nequi/Yape: "Превысил лимит"** → Сообщить лимиты метода, предложить банковский перевод для крупных сумм
- **Обменный курс** → PSP конвертирует из локальной валюты в USD. Клиент получает сумму в USD после конвертации

### 2.3 Уведомления при получении депозита

| Сумма | Кого уведомлять | Канал | Требуемое действие |
|-------|---------------------|-------|------------------|
| Любая сумма | Автоматическая система | Skale + MT5 | Автоматическое зачисление |
| > $5,000 | Finance + Pepe (Dealing) | Slack #deposits | Finance верифицирует, Pepe корректирует book при необходимости |
| > $10,000 | Finance + Susana (Compliance) + Pepe | Slack #deposits + email | Триггер EDD (см. раздел 5) |
| > $25,000 | Все вышеперечисленные + Дирекция | Slack + email + WhatsApp | Явное одобрение Compliance до зачисления |

### 2.4 Когда депозит не отображается — Troubleshooting для Support

**Скрипт для агента поддержки (Intercom):**

```
Шаг 1: Запросить у клиента:
   - Использованный метод оплаты
   - Сумму и валюту
   - Дату и время транзакции
   - Подтверждение оплаты (скриншот или TX hash)
   - Номер счёта MT5

Шаг 2: Проверить в Skale CRM:
   - Найти клиента по email или номеру счёта
   - Перейти в раздел "Deposits"
   - Проверить, есть ли ожидающий, неудачный или завершённый депозит

Шаг 3: По статусу:
   - "Completed" но не в MT5 → Эскалировать в Finance: "Депозит завершён в Skale, но не отражён в MT5"
   - "Pending" → Информировать клиента, что в обработке, указать сроки по методу
   - "Failed" → Информировать клиента, предложить повторить или использовать другой метод
   - Запись не существует → Возможно webhook не дошёл. Эскалировать в Finance с подтверждением клиента

Шаг 4: Если не решено за 2 часа (мгновенные) или 48 часов (wire):
   → Эскалировать в Finance с тикетом Intercom
   → Finance связывается с PSP для отслеживания платежа
```

## 3. Процесс вывода — пошагово

### 3.1 Общий поток

```
Клиент запрашивает вывод в Личном кабинете
        |
        v
Система проверяет: доступные средства, свободная маржа, открытые позиции
        |
        v
[Если проверка не пройдена] → Автоматический отказ с указанием причины
        |
        v
Заявка попадает в очередь одобрения в Skale
        |
        v
┌─────────────────────────────────────────────────┐
│            УРОВНИ ОДОБРЕНИЯ                      │
│                                                  │
│  < $1,000         → Авто-одобрение (если AML ок)│
│  $1,000 - $5,000  → Finance одобряет            │
│  $5,000 - $10,000 → Finance + Susana проверяют  │
│  > $10,000        → Finance + Susana одобряют    │
│                     + Pepe уведомлён             │
│  > $50,000        → Все + Дирекция              │
└─────────────────────────────────────────────────┘
        |
        v
Автоматические AML-проверки:
  - Верификация KYC завершена
  - Правило того же метода (same-method rule)
  - Проверка минимального объёма торговли
  - Проверка недавних депозитов (cooling period для карт)
        |
        v
[Если пройдено] → Finance обрабатывает вывод через PSP
        |
        v
Средства отправлены клиенту
        |
        v
Email-подтверждение клиенту
```

### 3.2 Правило того же метода (Same-Method Rule)

**Обязательно по регулированию AML:**

Клиент ОБЯЗАН выводить тем же методом, что использовал для пополнения, до суммы, внесённой этим методом.

**Пример:**
- Клиент внёс $3,000 через Visa и $2,000 через USDT
- Запрашивает вывод $4,000
- Первые $3,000 идут обратно на Visa
- Оставшиеся $1,000 идут на USDT
- Если есть дополнительная прибыль, клиент может выбрать метод

**Исключения:**
- Карта просрочена или отменена → Запросить подтверждение от банка + вывод через wire transfer
- PSP больше не работает с этим методом → Вывод через wire transfer с одобрения Compliance
- Депозит был через метод только для пополнения (OXXO, Western Union) → Вывод через wire transfer на имя владельца счёта

### 3.3 Сроки обработки по методам

| Метод | Время обработки | Время до получения клиентом |
|--------|------------------------|----------------------------------|
| Visa/Mastercard | 1 рабочий день | 3-5 рабочих дней (refund на карту) |
| Wire Transfer | 1 рабочий день | 2-5 рабочих дней |
| USDT TRC-20 | В тот же день | 5-30 минут после отправки |
| USDT ERC-20 | В тот же день | 5-30 минут после отправки |
| Bitcoin | В тот же день | 10-60 минут после отправки |
| PIX | 1 рабочий день | Мгновенно после обработки |
| PSE | 1 рабочий день | 1-2 рабочих дня |
| SPEI | 1 рабочий день | Минуты после обработки |
| Nequi | 1 рабочий день | Мгновенно после обработки |
| Mercado Pago | 1 рабочий день | 1-2 рабочих дня |
| Skrill/Neteller | 1 рабочий день | Мгновенно после обработки |

**Примечание:** "Время обработки" = от внутреннего одобрения до момента, когда Finance выполнит отправку. Выводы обрабатываются пакетами 2 раза в день: 10:00 и 16:00 (время сервера).

### 3.4 Автоматические AML-проверки при выводах

Перед одобрением вывода система (Skale + ручная проверка) должна верифицировать:

- [ ] **KYC завершён:** Личность верифицирована, подтверждение адреса, селфи с картой (если применимо)
- [ ] **Same-method rule соблюдено:** Вывод на тот же метод пополнения
- [ ] **Cooling period для карт:** Минимум 5 рабочих дней с момента пополнения картой до вывода (предотвращение chargebacks)
- [ ] **Минимальный объём торговли:** Клиент должен проторговать минимум 1 стандартный лот на каждые $5,000 депозита (предотвращение простого отмывания через пополнение-вывод)
- [ ] **Нет AML-алертов в профиле:** Без предыдущих флагов compliance
- [ ] **Совпадение имён:** Получатель вывода совпадает с владельцем счёта

### 3.5 Крипто-выводы — Специфический процесс

1. Клиент вводит адрес кошелька на портале
2. **Верификация кошелька:** Система проверяет, что кошелёк не в санкционных списках (OFAC и др.) — использовать инструмент скрининга типа Chainalysis или аналог
3. Finance верифицирует сумму и одобряет
4. Finance выполняет отправку с кошелька компании
5. **КРИТИЧНО:** Всегда проверять адрес кошелька 3 раза перед отправкой. Крипто-транзакции необратимы
6. Зафиксировать TX hash в Skale как подтверждение
7. Отправить TX hash клиенту по email

**Безопасность кошельков компании:**
- Кошелёк для выводов = "hot wallet" с ограниченными средствами (максимум $50,000)
- Крупные средства в "cold wallet" (аппаратный кошелёк) с доступом только Finance Manager + Susana (двойная подпись)
- Пополнение hot wallet = по мере необходимости, одобряется Finance Manager

### 3.6 Когда у вывода проблемы — Troubleshooting

**"Мой вывод был отклонён":**
- Проверить причину в Skale (недостаточно средств, KYC не завершён, открытые позиции, same-method rule)
- Сообщить клиенту конкретную причину и как её устранить

**"Мой вывод одобрен, но не приходит":**
- Проверить в Skale, обработан ли через PSP
- Если крипто: проверить TX hash в блокчейне
- Если карта: сообщить, что refund на карту занимает 3-5 рабочих дней
- Если wire: попросить клиента обратиться в банк, предоставить подтверждение SWIFT если доступно

**"Хочу вывести другим методом":**
- Объяснить same-method rule
- Если есть обоснованное исключение, эскалировать в Compliance (Susana) для одобрения

## 4. Ежедневная сверка

### 4.1 Что это и почему критично

Ежедневная сверка сопоставляет 3 источника данных для обеспечения учёта каждого цента:

1. **PSP (Payment Service Providers):** Что платёжные процессоры отчитывают как внесённое/выведенное
2. **Skale CRM:** Что наша система зарегистрировала как внесённое/выведенное
3. **MT5:** Что клиенты видят на своих торговых счетах

Если эти 3 не совпадают, есть проблема: незачисленный депозит, дублирование вывода, мошенничество или системная ошибка.

### 4.2 Ежедневный чеклист сверки

**Рекомендуемое время:** 9:00 утра, до обработки выводов за день

```
ЕЖЕДНЕВНАЯ СВЕРКА — NEOMAAA
Дата: ___________
Ответственный: ___________

ШАГ 1: PSP vs Skale
─────────────────────
[ ] Скачать отчёт о депозитах за предыдущий день из КАЖДОГО PSP
[ ] Скачать отчёт о депозитах из Skale CRM за предыдущий день
[ ] Сверить транзакцию за транзакцией:
    - Каждый депозит в PSP должен существовать в Skale
    - Каждый депозит в Skale должен существовать в PSP
    - Суммы должны совпадать (с учётом комиссий PSP)
[ ] Зафиксировать расхождения:
    - Депозиты в PSP, но не в Skale = webhook не прошёл → зачислить вручную
    - Депозиты в Skale, но не в PSP = возможная системная ошибка → расследовать

ШАГ 2: Skale vs MT5
─────────────────────
[ ] Сгенерировать отчёт о начислениях/списаниях в MT5 за предыдущий день
[ ] Сверить со Skale:
    - Каждый "завершённый" депозит в Skale должен иметь зачисление в MT5
    - Каждый "обработанный" вывод в Skale должен иметь списание в MT5
[ ] Зафиксировать расхождения:
    - Депозит в Skale без зачисления в MT5 = сбой интеграции → ручное зачисление в MT5
    - Зачисление в MT5 без депозита в Skale = возможное несанкционированное зачисление → расследовать СРОЧНО

ШАГ 3: Общие балансы
─────────────────────────
[ ] Просуммировать общие депозиты за день (по PSP)
[ ] Просуммировать общие выводы за день (по PSP)
[ ] Проверить, что нетто-баланс дня совпадает с движением на банковском счёте / кошельках
[ ] Сравнить баланс крипто-кошелька компании с ожидаемым

ШАГ 4: Ожидающие выводы
───────────────────────────
[ ] Перечислить все выводы, ожидающие одобрения
[ ] Проверить, что ни один вывод не в обработке более 48 часов
[ ] Приоритизировать выводы с ближайшими дедлайнами

ШАГ 5: Красные флаги
──────────────────
[ ] Проверить дублирующиеся депозиты (один клиент, та же сумма, интервал в минуты)
[ ] Проверить выводы, превышающие общий депозит клиента
[ ] Проверить клиентов с депозитом + немедленным выводом без торговли
[ ] Проверить транзакции на повторяющиеся "круглые" суммы ($999, $4,999 — structuring)

РЕЗУЛЬТАТ:
[ ] Всё сходится — Подписать и архивировать
[ ] Есть расхождения — Детализировать и эскалировать

Примечания: _________________________________
Подпись: _________________________________
```

### 4.3 Критические красные флаги

| Красный флаг | Немедленное действие |
|----------|------------------|
| Депозит в PSP, не существующий в Skale | Ручное зачисление клиенту. Расследовать webhook |
| Зачисление в MT5 без соответствующего депозита | ЗАМОРОЗИТЬ счёт. Расследовать несанкционированный доступ |
| Вывод обработан дважды | Немедленно связаться с PSP для отмены дубликата |
| Баланс крипто-кошелька не сходится | Заморозить крипто-выводы. Аудит всех транзакций |
| Клиент с депозитом-выводом без торговли | AML флаг → Эскалировать Susana |
| Множественные депозиты чуть ниже $10,000 | Подозрение на structuring → Эскалировать Susana |

## 5. Протокол крупных транзакций

### 5.1 Пороги Enhanced Due Diligence (EDD)

| Триггер | Порог | Действие |
|---------|--------|--------|
| Крупный индивидуальный депозит | > $10,000 | EDD требуется до зачисления |
| Накопительный месячный депозит | > $25,000 за месяц | EDD требуется |
| Крупный вывод | > $10,000 | EDD + одобрение Compliance |
| Новый клиент с крупным депозитом | > $5,000 в первую неделю | EDD требуется |
| Внезапное изменение паттерна | Депозит в 3 раза больше среднего | EDD требуется |

### 5.2 Требуемая документация (Source of Funds)

При активации EDD запросить у клиента:

```
ТРЕБУЕМАЯ ДОКУМЕНТАЦИЯ — ПРОИСХОЖДЕНИЕ СРЕДСТВ

Уважаемый/ая [имя клиента],

Для обработки вашей транзакции на сумму [сумма] нам необходимо 
подтвердить происхождение средств в соответствии с нашими 
регуляторными обязательствами.

Пожалуйста, предоставьте ОДНО из следующего:

□ Недавняя квитанция о зарплате (за последние 3 месяца)
□ Налоговая декларация за последний год
□ Банковская выписка, показывающая источник средств (за последние 3 месяца)
□ Документ о продаже недвижимости или транспортного средства
□ Документ о наследстве или дарении
□ Трудовой договор или справка с работы с указанием зарплаты
□ Выписка с инвестиционного счёта (если средства от другой инвестиции)

ДОПОЛНИТЕЛЬНО для сумм > $25,000:
□ Подписанная декларация о происхождении средств (форма предоставлена NEOMAAA)
□ Подтверждение перевода с исходного счёта

У вас 7 календарных дней для предоставления этой документации.
Если мы не получим её, средства будут возвращены оригинальным 
методом пополнения.

Благодарим за понимание.
```

### 5.3 Поток одобрения крупных транзакций

```
Обнаружен депозит/вывод > $10,000
        |
        v
Finance получает автоматическое уведомление (Slack #large-transactions)
        |
        v
Finance запрашивает документацию SOF у клиента (через Intercom или email)
        |
        v
Клиент предоставляет документацию
        |
        v
Finance проверяет документацию (соответствует ли профилю клиента?)
        |
        v
Finance передаёт документацию Susana (Compliance) с оценкой
        |
        v
Susana проверяет и решает:
  ├─ ОДОБРЕНО → Finance приступает к транзакции
  ├─ НУЖНА ДОИНФОРМАЦИЯ → Finance запрашивает дополнительную документацию у клиента
  └─ ОТКЛОНЕНО → Finance возвращает средства клиенту + Susana регистрирует SAR при необходимости
        |
        v
Pepe (Dealing) уведомлён о сумме для управления книгой
```

**Сроки:**
- Finance должен проверить документацию максимум за 24 часа с момента получения
- Susana должна одобрить/отклонить максимум за 48 часов
- Клиент должен получать обновление статуса каждые 24 часа
- Если клиент не предоставляет документацию за 7 дней, вернуть средства

### 5.4 Влияние на Dealing (Pepe)

Pepe должен знать о крупных депозитах потому что:
- Депозит $50,000+ может генерировать позиции, влияющие на нетто-экспозицию брокера
- Pepe должен подготовить хеджирование, если у клиента история крупных позиций
- Если клиент — кандидат на "A-book" (последовательный трейдер с высоким объёмом), Pepe может перевести клиента на A-book до открытия позиций

**Уведомление Pepe:**
- Депозиты > $5,000: Информационное уведомление
- Депозиты > $10,000: Pepe должен проверить торговый профиль клиента и подготовить план управления рисками
- Депозиты > $50,000: Краткое совещание (5 мин) между Finance, Pepe и Susana до зачисления

## 6. Предотвращение мошенничества

### 6.1 Chargebacks (чарджбэки по картам)

**Чарджбэк — главный финансовый риск для депозитов картами.**

#### Предотвращение:
- 3D Secure обязателен для всех депозитов картой
- Cooling period 5 рабочих дней до разрешения вывода средств, внесённых картой
- Селфи с картой требуется для депозитов картой > $500
- Дневной лимит $10,000 на карту
- Максимум 3 разные карты на счёт

#### При получении чарджбэка:

```
ПРОТОКОЛ ЧАРДЖБЭКА

1. PSP уведомляет о чарджбэке через email/дашборд
   Ответственный: Finance
   Время: Немедленно при получении

2. Finance замораживает счёт клиента
   - Отключить торговлю
   - Отключить выводы
   - Отключить депозиты

3. Finance собирает доказательства для оспаривания:
   - Копия KYC клиента (ID, подтверждение адреса)
   - Лог IP сессии, в которой был сделан депозит
   - История торговли (доказывает, что услуга была оказана)
   - Предыдущие коммуникации с клиентом
   - Подтверждение 3D Secure аутентификации
   - Селфи с картой если есть

4. Finance отправляет representment (оспаривание) в PSP в течение 5 рабочих дней

5. Susana регистрирует инцидент:
   - В профиле клиента в Skale
   - В реестре чарджбэков
   - Оценивает, есть ли паттерн (множественные чарджбэки = организованное мошенничество)

6. Если чарджбэк проигран:
   - Убыток регистрируется бухгалтерски
   - Клиент ЗАБЛОКИРОВАН навсегда
   - Добавить в внутренний чёрный список (имя, email, IP, карта)

7. Если чарджбэк выигран:
   - Разморозить счёт клиента
   - Усиленный мониторинг 90 дней
   - Рассмотреть ограничение методов оплаты только крипто/wire
```

### 6.2 Индикаторы украденной карты

| Индикатор | Уровень риска | Действие |
|-----------|----------------|--------|
| IP депозита в стране, отличной от страны карты | Высокий | Заблокировать депозит, запросить верификацию |
| Множественные карты на одном счёте за короткое время | Высокий | Заморозить счёт, расследовать |
| Имя на карте не совпадает с именем на счёте | Критический | Отклонить депозит немедленно |
| Депозит с последующим немедленным выводом в крипто | Критический | Заморозить счёт + эскалировать в Compliance |
| Множественные неудачные попытки перед успешной | Средний | Запросить селфи с картой |
| Обнаружен VPN или прокси | Средний | Запросить дополнительную верификацию |
| Одноразовый email (mailinator, guerrillamail и т.д.) | Средний | Отклонить регистрацию |

### 6.3 Злоупотребление мульти-аккаунтами

**Признаки тревоги:**
- Один и тот же отпечаток устройства/браузера на нескольких счетах
- Один IP на нескольких счетах
- Похожие имена или один адрес на нескольких счетах
- Одна карта или крипто-кошелёк на нескольких счетах
- Идентичный торговый паттерн на нескольких счетах (копирование самого себя)

**Действия:**
1. Обнаружить через Skale CRM (сопоставить данные клиентов)
2. Заморозить все подозрительные счета
3. Расследовать: это одно лицо? Есть ли мотив злоупотребления бонусами?
4. Если подтверждено: закрыть дополнительные счета, оставить только 1
5. При мошенническом умысле: закрыть все, вернуть нетто-средства, заблокировать

### 6.4 Обнаружение structuring (smurfing)

**Structuring = Разбиение крупных транзакций на множество мелких для обхода порогов отчётности.**

**Паттерны для мониторинга:**
- Множественные депозиты $9,000-$9,999 (чуть ниже $10,000)
- 3+ депозита в один день от одного клиента
- Депозиты на повторяющиеся круглые суммы ($5,000, $5,000, $5,000...)
- Клиент часто пополняет, но не торгует (или торгует минимально)
- Депозиты разными методами оплаты за короткий период

**Действие при обнаружении structuring:**
1. Finance помечает счёт для проверки
2. Susana проводит полное AML-расследование
3. При обоснованном подозрении: подать SAR (Suspicious Activity Report) в соответствующий орган (Anjouan)
4. Не сообщать клиенту о расследовании (tipping off незаконен)
5. В зависимости от оценки: заморозить счёт или разрешить операции под усиленным мониторингом

| Риск | Влияние | Смягчение |
|--------|---------|------------|

| Недостаток финансовой экспертизы | Ошибки в сверке, потеря средств | Создать детальные чеклисты (данный документ). Дирекция проверяет еженедельно |

### 8.4 Упрощённая сверка (промежуточная версия)

Пока нет Finance Manager, Susana выполняет сокращённую версию:

```
УПРОЩЁННАЯ ЕЖЕДНЕВНАЯ СВЕРКА (15-20 минут)
──────────────────────────────────────────────────

1. [ ] Открыть дашборд каждого PSP
2. [ ] Проверить, что сумма депозитов за предыдущий день в PSP = сумме в Skale
3. [ ] Проверить, что сумма выводов за предыдущий день в PSP = сумме в Skale
4. [ ] Если суммы сходятся (разница < $50): ОК, подписать и архивировать
5. [ ] Если не сходятся: выявить расхождение и эскалировать Дирекции
6. [ ] Проверить список выводов в ожидании > 24 часов
7. [ ] Проверить баланс hot wallet крипто

День: __________  Сходится: ДА / НЕТ  Подпись: __________
```

**Полная еженедельная сверка (пятница):**
Дирекция выполняет полную сверку по чеклисту из Раздела 4.2, покрывая всю неделю.

### 8.5 Что должен сделать Finance Manager при найме

**Первые 2 недели:**
1. Получить данный документ и доступы к PSP, Skale, MT5 Admin
2. Выполнить полную сверку за последние 30 дней (выявить накопившиеся расхождения)
3. Принять на себя все задачи из таблицы 8.2
4. Установить прямую связь с каждым PSP (контактные лица, SLA, эскалация)
5. Проверить и улучшить данный документ на основе реальных операций

**Первый месяц:**
6. Внедрить автоматизированный дашборд сверки (идеал: подключить PSP API к Skale через скрипт)
7. Установить еженедельные и ежемесячные финансовые отчёты
8. Оптимизировать сроки обработки выводов
9. Договориться о лучших тарифах с PSP на основе объёма

### 9.6 Матрица быстрого решения — Выводы

```
KYC завершён?
  НЕТ → Отклонить вывод. Попросить клиента завершить KYC.
  ДА ↓

Вывод соответствует правилу same-method?
  НЕТ → Есть обоснованное исключение? (карта просрочена, метод недоступен)
       НЕТ → Отклонить. Сообщить клиенту о необходимости вывода тем же методом.
       ДА → Эскалировать в Compliance для одобрения исключения.
  ДА ↓

Сумма < $1,000?
  ДА → Авто-одобрение (если нет AML-флагов).
  НЕТ ↓

Сумма между $1,000 и $5,000?
  ДА → Finance одобряет (проверить базовую историю).
  НЕТ ↓

Сумма между $5,000 и $10,000?
  ДА → Finance + Compliance проверяют. EDD при необходимости.
  НЕТ ↓

Сумма > $10,000?
  ДА → Finance + Compliance одобряют. EDD обязателен. Уведомить Pepe.
       > $50,000? → Дополнительно, одобрение Дирекции.
```

## Приложение: Глоссарий

| Термин | Определение |
|---------|-----------|
| **PSP** | Payment Service Provider — Компания, обрабатывающая платежи (напр.: процессор карт, крипто-шлюз) |
| **KYC** | Know Your Customer — Верификация личности клиента |
| **AML** | Anti-Money Laundering — Предотвращение отмывания денег |
| **EDD** | Enhanced Due Diligence — Усиленная проверка для транзакций высокого риска |
| **SOF** | Source of Funds — Происхождение средств |
| **SAR** | Suspicious Activity Report — Отчёт о подозрительной активности в регуляторный орган |
| **Same-method rule** | Обязательство выводить средства тем же методом, что использовался для пополнения |
| **Cooling period** | Период ожидания между депозитом и выводом (предотвращение чарджбэков) |
| **Structuring/Smurfing** | Разбиение крупных транзакций на мелкие для обхода порогов отчётности |
| **Hot wallet** | Крипто-кошелёк с доступными средствами для ежедневных операций |
| **Cold wallet** | Офлайн крипто-кошелёк для безопасного хранения крупных сумм |
| **Representment** | Процесс оспаривания чарджбэка с представлением доказательств |
| **A-book** | Модель, при которой операции клиента направляются напрямую на рынок |
| **B-book** | Модель, при которой брокер принимает на себя риск, противоположный позиции клиента |
